# Domain Model and Server Contracts

Этот документ фиксирует **доменную модель платформы**, целевые серверные контракты и базовые принципы отказоустойчивости. Он служит опорной спецификацией для реализации backend-модулей, административной панели и ручной матрицы тестирования.

## Архитектурная рамка

Платформа делится на несколько связанных контуров. Контур **управления доступом** отвечает за пользователей, роли, API-ключи и подписки. Контур **операционного выполнения** отвечает за задания, очереди, воркеры, lease-модель прокси и execution events. Контур **финансов и биллинга** отвечает за тарифы, usage metering, платежи и контроль себестоимости. Контур **наблюдаемости и аудита** отвечает за телеметрию, health checks, историю действий и системные настройки.

| Контур | Основные сущности | Назначение |
| --- | --- | --- |
| Access | users, apiKeys, subscriptions, plans | Управление идентичностью, ролями, лимитами и тарифами |
| Execution | jobs, jobEvents, workerNodes, workerRuns | Оркестрация заданий, статусы, retries и журнал выполнения |
| Proxy | proxyProviders, proxyPolicies, proxyLeases | Управление провайдерами, sticky/rotating-сессиями, fallback и cost tracking |
| Billing | payments, usageRecords | Учёт потребления, стоимости, статусов платежей и подписок |
| Observability | metricSnapshots, auditTrail, systemSettings, telegramEndpoints | Метрики, аудит, настройки системы и уведомления |

## Доменная модель

Сущность **users** остаётся корневой для авторизации и ролевого доступа. Пользователь может владеть несколькими API-ключами, подписками, платежами, заданиями и настройками уведомлений. Роль `admin` получает доступ к полному интерфейсу управления платформой, а роль `user` ограничивается своими заданиями, ключами и подписками.

Сущность **plans** задаёт тарифный контракт: частоту биллинга, квоты, RPM, конкурентность и наличие VIP-доступа. Сущность **subscriptions** связывает пользователя с тарифом и хранит провайдера платежей, период действия и внешние идентификаторы биллинга. Сущность **payments** фиксирует отдельные платежные события, чтобы отделить жизненный цикл подписки от жизненного цикла конкретной транзакции.

Сущность **apiKeys** описывает машиночитаемый канал доступа к REST API. Для каждого ключа отдельно фиксируются scope, статус, ограничения по запросам, сроки действия и следы использования. Это нужно как для rate limiting, так и для раздельного контроля стоимости single, bulk и VIP-контуров.

Сущность **jobs** становится центральной единицей исполнения. Каждое задание содержит источник запуска, режим запроса, payload, статус, retry-параметры, оценку стоимости и итоговую себестоимость. Сущность **jobEvents** даёт подробную ленту событий по заданию, позволяя видеть переходы статусов, технические ошибки, смену прокси-провайдера и поведение retry-логики.

Сущности **workerNodes** и **workerRuns** отделяют логическую модель очереди от реального исполнения. Узел воркера хранит состояние здоровья, конкурентность и heartbeat. Запуск воркера хранит конкретную попытку исполнения, профиль политики, отпечаток исполнения и длительность выполнения. Это позволяет делать ручной разбор ошибок не на уровне абстрактного job, а на уровне отдельных run-attempt.

Сущности **proxyProviders**, **proxyPolicies** и **proxyLeases** формируют отдельный proxy-layer. Провайдер описывает коммерческие и технические параметры источника трафика. Политика задаёт желаемое поведение маршрутизации, sticky/rotating-модель, географические ограничения и пределы переключения. Lease описывает конкретно выданную прокси-сессию, её срок жизни, объём переданных байтов, estimated cost и связанные ошибки транспорта.

Сущность **usageRecords** нужна для тарификации и метеринга. Она связывает пользователя, API-ключ, job и тип потребления. Это даёт возможность одновременно считать квоты, стоимость, COGS и бизнесовую выручку. Сущность **metricSnapshots** нужна для агрегированных срезов по системе, очередям, провайдерам и биллингу. Сущность **auditTrail** хранит все значимые действия пользователя, API-ключа, системного процесса или воркера и является базой для расследования инцидентов. Сущность **systemSettings** хранит централизованные конфигурации системы, а **telegramEndpoints** связывает систему с каналами уведомлений.

## Серверные контракты

Сервер делится на два слоя. Внутренний слой реализуется через typed procedures для административной панели и внутренних экранов. Внешний слой — через REST-маршруты для API-клиентов. Эти два слоя используют общую доменную модель и одни и те же бизнес-правила.

### REST API

| Endpoint | Метод | Назначение | Аутентификация |
| --- | --- | --- | --- |
| `/api/v1/requests/single` | POST | Создание одиночного задания | API key |
| `/api/v1/requests/bulk` | POST | Создание пакетного задания | API key |
| `/api/v1/requests/vip` | POST | Создание приоритетного VIP-задания | API key с VIP scope |
| `/api/v1/jobs/:publicId` | GET | Получение статуса и результата задания | API key или владелец |
| `/api/v1/jobs/:publicId/events` | GET | Получение ленты событий задания | API key или admin |
| `/api/v1/usage/summary` | GET | Агрегированный usage и cost | API key или admin |
| `/api/v1/health` | GET | Health summary публичного контура | Публично или service token |

REST-контракт должен поддерживать единый формат ответа. Успешный ответ возвращает `ok`, `data`, `meta` и `requestId`. Ошибочный ответ возвращает `ok: false`, `error.code`, `error.message`, `error.retryable` и `requestId`. Такое выравнивание позволит потом строить надёжные ручные тесты и единый error-handling в UI.

## Принципы отказоустойчивости

Основная задача стабильности состоит не только в обработке исключений, но и в том, чтобы **каждое состояние было наблюдаемым и воспроизводимым**. Поэтому каждый job должен иметь чёткую ленту событий, каждый transport error должен классифицироваться, а каждое изменение конфигурации должно отражаться в audit trail.

| Риск | Контрмера |
| --- | --- |
| Падение конкретного прокси-провайдера | Provider health score, fallback-переключение и job event о смене провайдера |
| Повторяющиеся transport errors | Retry-policy с лимитами и классификацией retryable/non-retryable |
| Перегрузка очереди | Queue depth monitoring, concurrency limits, приоритизация VIP-jobs |
| Ошибки конфигурации | Централизованные system settings и audit events на каждое изменение |
| Рост себестоимости | Usage metering, snapshot по COGS, cost ceiling на proxy policy |
| Непрозрачные инциденты | Полный audit trail и детализированные job events |

## Подход к ручным тестам

Ручные тесты будут строиться не вокруг произвольных кликов, а вокруг **критических пользовательских и операционных сценариев**. Это означает, что для каждой зоны платформы будет отдельная серия проверок: создание задания, смена статуса, ограничение по API-ключу, simulated proxy fallback, simulated worker retry, запись usage, изменение подписки, генерация audit event, отправка тестового Telegram-события и отображение агрегатов на панели.

Перед началом ручных прогонов backend должен сначала получить минимально работоспособный каркас с моковыми и безопасными сценариями, чтобы тестирование было повторяемым и не зависело от внешних защищённых систем. Именно на этот каркас затем будут наложены unit/integration tests и журнал ошибок с последующей стабилизацией.
