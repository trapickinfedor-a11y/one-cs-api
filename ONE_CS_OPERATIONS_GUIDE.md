# ONE CS Operations Guide

## Назначение документа

Этот документ фиксирует рабочую логику **ONE CS** в текущем проекте, чтобы оператор, администратор и технический специалист одинаково понимали, как интерпретировать результаты проверки, что означает каждая оценка и в каком порядке использовать интерфейс системы.

Система сейчас строит итог по трём основным слоям. Сначала определяется **raw credit score** в диапазоне **300–850** либо фиксируется отсутствие кредитного файла. Затем из него формируется **product score** по шкале **1–20**. После этого рассчитывается **data quality score** по шкале **1–10** с учётом полноты анкеты и adverse reasons. Именно `dataQualityScore` является основной интегральной оценкой качества профиля для операторского чтения.

## Итоговый контракт результата ONE CS

Каждый итоговый результат ONE CS в безопасном и рабочем контуре должен интерпретироваться по следующей модели.

| Поле | Формат | Смысл |
|---|---:|---|
| `creditScore` | `300–850` или `null` | Сырой credit score, если он найден |
| `productScore` | `1–20` | Нормализованная продуктовая шкала для быстрого сравнения результатов |
| `dataQualityScore` | `1–10` | Интегральная оценка качества данных и кредитного профиля |
| `status` | `success`, `review`, `decline`, `no_file`, `error` | Операционный статус результата |
| `adverseReasons` | массив строк | Нормализованные adverse reasons, влияющие на итог |
| `adverseReasonGroups` | массив групп | Категории adverse reasons для логики штрафов |
| `priceUsd` | число | Расчётная цена проверки |
| `durationMs` | целое число | Длительность обработки |
| `source` | строка | Источник запуска: dashboard, api, import, testbench и т.д. |
| `explanations` | массив строк | Готовые текстовые объяснения результата для оператора |
| `completenessScore` | `0–1`, опционально | Внутренняя оценка полноты входных данных |

## Как читается каждая шкала

### Raw credit score 300–850

Это исходный кредитный показатель. Он сам по себе не равен финальному качеству данных, но задаёт базовый уровень доверия к профилю. Чем ниже значение, тем ниже стартовая база для дальнейшего расчёта.

### Product score 1–20

`productScore` — это линейно нормализованная шкала, которая помогает быстро сравнивать результаты в интерфейсе. Она удобна для визуального ранжирования, но не должна использоваться как единственный критерий решения. Для операционной интерпретации всегда нужно смотреть на `dataQualityScore`, статус и adverse reasons одновременно.

### Data quality score 1–10

`dataQualityScore` — это главная итоговая оценка. Она создаётся из трёх блоков: базовой credit-score полосы, поправки на полноту входных данных и штрафов за adverse reasons. Если credit file отсутствует, система стартует с низкой базы. Если анкета полная и adverse signals отсутствуют, итог растёт. Если обнаружены признаки thin file, affordability pressure, delinquency, public records или перегрузка inquiries, итог снижается.

## Правила интерпретации `status`

| Status | Когда присваивается | Как трактовать оператору |
|---|---|---|
| `success` | `dataQualityScore >= 7.5` | Профиль выглядит сильным и достаточно качественным для позитивного чтения |
| `review` | `dataQualityScore >= 4.5` и `< 7.5` | Профиль пограничный; требуется ручная оценка и чтение причин |
| `decline` | `dataQualityScore < 4.5` | Профиль слабый либо содержит тяжёлые adverse signals |
| `no_file` | credit file не найден и есть группа `no_file` | Кредитный файл не обнаружен; автоматическая уверенная интерпретация невозможна |
| `error` | техническая ошибка | Результат не должен трактоваться как бизнес-решение |

## Логика `dataQualityScore 1–10`

Текущая модель использует три слоя.

### 1. Базовое качество по credit score

| Credit score | База качества |
|---|---:|
| `>= 800` | `9.7` |
| `760–799` | `9.0` |
| `720–759` | `8.2` |
| `680–719` | `7.4` |
| `640–679` | `6.5` |
| `600–639` | `5.5` |
| `560–599` | `4.5` |
| `520–559` | `3.5` |
| `480–519` | `2.5` |
| `< 480` | `1.5` |
| `null` | `2.0` |

### 2. Поправка на полноту анкеты

`completenessScore` остаётся внутренним сигналом в диапазоне `0–1`. Он не заменяет `dataQualityScore`, а корректирует его.

| Completeness score | Поправка |
|---|---:|
| `>= 0.95` | `+0.8` |
| `>= 0.80` | `+0.5` |
| `>= 0.65` | `+0.2` |
| `>= 0.50` | `0.0` |
| `>= 0.35` | `-0.5` |
| `< 0.35` | `-1.0` |

### 3. Штрафы по adverse reason groups

Система нормализует adverse reasons в типовые группы и затем суммирует штрафы. Суммарный штраф ограничивается сверху, чтобы даже очень тяжёлый профиль не уходил ниже минимальной границы шкалы.

| Группа | Примеры причин | Штраф |
|---|---|---:|
| `no_file` | Unable to find credit profile at TransUnion | `5.0` |
| `thin_file` | Insufficient credit history; Insufficient length of credit history | `2.2` |
| `low_depth` | Lack of recent installment loan information | `1.4` |
| `affordability` | Income or credit history insufficient for loan; Requested amount unsupported by income; High debt in relation to income | `2.0` |
| `utilization` | Too many accounts with balances; Proportion of balances to credit limits is too high | `1.6` |
| `delinquency` | Serious delinquency; Time since delinquency is too recent or unknown | `2.4` |
| `public_record` | Serious delinquency, and public record or collection filed | `3.2` |
| `inquiry_pressure` | RiskView Consumer Inquiry; Too many inquiries last 12 months | `1.0` |
| `consumer_finance` | Too many consumer finance company accounts | `0.8` |

### Финальная формула

Итоговая оценка считается как:

> `dataQualityScore = clamp(1, 10, baseQuality + completenessAdjustment + bonus - penalty)`

Небольшой бонус применяется только в сильной зоне, когда adverse reasons отсутствуют и score находится в хорошем диапазоне. Финальное значение округляется до одной цифры после запятой.

## Как трактовать adverse reasons из запроса пользователя

Причины вроде **Insufficient length of credit history**, **Income or credit history insufficient for loan** и **Requested amount unsupported by income** уже должны рассматриваться не как отдельные текстовые сообщения, а как бизнес-сигналы, влияющие на единую оценку `dataQualityScore 1–10`.

| Причина | Группа | Практический смысл |
|---|---|---|
| `Insufficient length of credit history` | `thin_file` | У профиля слишком короткая кредитная история; уверенность снижается |
| `Income or credit history insufficient for loan` | `affordability` | Профиль выглядит недостаточным по доходу и/или глубине кредитной истории |
| `Requested amount unsupported by income` | `affordability` | Запрашиваемая нагрузка выглядит несоразмерной доходу |

Если эти причины появляются одновременно, оператор должен трактовать это как накопленный негативный сигнал, а не как три независимые текстовые строки. В интерфейсе это должно сводиться к понижению `dataQualityScore`, статусу `review` либо `decline` и набору объяснений в `explanations`.

## Рабочий порядок для оператора

Оператор должен смотреть на результат в одном и том же порядке. Сначала нужно проверить, найден ли `creditScore`. Затем нужно посмотреть на `productScore` как на быстрый ориентир. После этого нужно читать `dataQualityScore` как главный итог. Далее необходимо просмотреть `status`, adverse reasons и explanations. Если статус равен `review`, решение нельзя принимать только по числу — нужно обязательно читать причины и оценивать полноту профиля.

## Правила быстрого чтения результатов

| Ситуация | Что делать |
|---|---|
| `success` и `dataQualityScore >= 8` | Профиль выглядит качественным; причины либо отсутствуют, либо слабо влияют на итог |
| `review` и есть `affordability` | Проверить доходную часть, размер запроса и полноту анкеты |
| `review` и есть `thin_file` | Проверить, не является ли профиль новым или неполным |
| `decline` и есть `public_record`/`delinquency` | Интерпретировать как тяжёлый негативный профиль |
| `no_file` | Не приравнивать к обычному decline; это отдельная ветка отсутствия файла |

## Где оператор видит эти данные

Сейчас контракт ONE CS должен быть доступен в основных административных и импортных контурах.

| Экран | Что видно |
|---|---|
| `Overview` | Краткий срез по последним jobs, включая credit score, product score и data quality |
| `Operations / Jobs` | Расширенное чтение результата, статуса и adverse reasons |
| `Imported Data` | Предпросмотр импортированных записей с вычисленным ONE CS результатом |
| REST / safe job responses | Полный `oneCsResult` в `resultJson` и summary |

## Эксплуатационные замечания

Система уже использует безопасный режим для preview и safe-batch контуров, поэтому при ручной проверке интерфейса оператор должен понимать, что часть jobs и событий может быть демонстрационной либо safe-mode симуляцией. Это не отменяет валидности скоринговой логики, но требует аккуратного чтения `source`, `status` и `safeTestMode`.

## Что считать завершённым перед production-использованием

Перед эксплуатацией на внешнем сервере необходимо подтвердить, что схема базы синхронизирована, тесты проходят, а dev-интерфейс без runtime-ошибок показывает итоговые поля ONE CS. Также нужно проверить, что инструкция по серверному запуску и окружению соответствует реальной конфигурации удалённого хоста.

## Короткий вывод

В текущей реализации **`dataQualityScore 1–10` уже является единой оценкой из credit score, completeness и adverse reasons**. Причины отказа или ослабления профиля не должны читаться отдельно от общего результата: они являются входами в интегральный скоринг, а итог оператор принимает по совокупности `dataQualityScore`, `status`, `adverseReasons` и `explanations`.
